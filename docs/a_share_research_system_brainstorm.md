# 个人 A 股中低频投资策略与风控研究系统设计（Brainstorm）

## 1. 目标与约束

### 核心目标
- 优先建设：
  - 投资策略研究能力
  - 风控策略研究能力
  - 研究验证与版本对比能力
- 通过历史数据做分版本研究，持续筛选更适配当前市场环境的策略与风控方案。

### 约束与定位
- 资金规模：50 万 ~ 500 万人民币。
- 标的与频率：A 股日线/周线中低频。
- 不做高频，不依赖盘口级流动性。
- 不追求复杂平台化，追求：高夏普、低回撤、可解释、可复现、可版本化。

---

## 2. 推荐总体架构（“轻平台 + 强研究”）

建议采用 5 层结构，避免过早工程化：

1. **数据层**：行情、财务、指数、行业、交易日历、复权与停牌处理。
2. **因子/特征层**：技术面、估值面、质量面、风险暴露、市场状态特征。
3. **研究与回测层**：策略生成、组合构建、交易规则、费用模型、滑点模型。
4. **风控层**：仓位控制、止损/止盈、风险预算、组合约束、市场状态降杠杆。
5. **评估与版本治理层**：实验管理、参数/代码/数据版本、报告与可复现运行。

---

## 3. 开源框架组合建议（按“够用优先”）

### 路线 A（推荐起步）：Backtrader + PyFolio/Empyrical + MLflow + DuckDB/Parquet
- **Backtrader**：中低频策略回测足够灵活，学习成本低，社区资料多。
- **Empyrical / PyFolio（或替代分析包）**：绩效与风险指标分析。
- **MLflow**：记录每次实验参数、指标、artifact（图表/报告）。
- **DuckDB + Parquet**：本地研究型数据湖，轻量高性能。

适合个人研究，工程复杂度低，快速形成“可版本化研究闭环”。

### 路线 B（进阶）：Qlib + 自定义回测执行层
- **Qlib** 在因子研究、数据处理、模型研究方面较强，适合后续扩展 ML 因子。
- 成本：框架学习与定制门槛更高，建议在基础流程跑顺后再迁移。

### 路线 C（国内生态）：vn.py（仅部分模块）+ 自建研究管线
- vn.py 强项在交易执行和事件驱动框架。
- 你当前目标偏“研究与验证”，可借鉴其事件模型，不必一上来全套接入。

---

## 4. 最小可行系统（MVP）分阶段建设

### Phase 0：研究基础设施（1~2 周）
- 建立目录规范：`data/`, `factors/`, `strategies/`, `risk/`, `experiments/`, `reports/`。
- 固定数据 schema（OHLCV、复权因子、交易状态、行业分类）。
- 建立统一回测输入输出接口（避免策略代码与数据处理耦合）。

### Phase 1：基线策略库（2~4 周）
先做 3 类可解释策略，形成对照组：
1. **趋势类**：均线趋势/突破（周线确认+日线执行）。
2. **均值回复类**：超跌反弹（加行业/市场状态过滤）。
3. **质量+估值轮动类**：季度/半年度调仓，低换手。

每个策略都要配标准风控模板，确保可横向比较。

### Phase 2：风控策略研究（2~4 周）
- 仓位层：固定仓位、波动率目标仓位、风险预算仓位。
- 组合层：最大单票权重、行业偏离上限、持仓数量下限。
- 交易层：涨跌停/停牌处理、成交约束、最小持有期、换手约束。
- 账户层：回撤分级降风险（如 5%/8%/12% 三档降仓）。

### Phase 3：版本化验证（持续）
- Walk-forward（滚动训练/验证）+ OOS（样本外）评估。
- Regime 分段评估（牛/熊/震荡，不同波动环境分桶）。
- 形成“策略版本卡”：适用环境、失效信号、风险边界。

## 5. 数据与回测关键细节（A 股特有）

1. **复权处理统一**：前复权用于信号，真实价格用于成交模拟（避免收益失真）。
2. **交易可行性约束**：
   - 涨跌停无法成交处理；
   - 停牌延迟成交；
   - ST、次新、低流动性过滤。
3. **费用模型真实化**：佣金 + 印花税 + 滑点分层（大盘/中盘/小盘）。
4. **生存偏差控制**：股票池按历史时点构建，避免“幸存者偏差”。
5. **时间点一致性**：财务数据和因子必须遵守“可得性时间戳”。

---

## 6. 风控研究框架（建议重点）

将风控拆为可插拔模块，独立版本化：

- `risk_position.py`：仓位管理
- `risk_entry.py`：开仓过滤
- `risk_exit.py`：止损止盈/移动止损
- `risk_portfolio.py`：组合约束
- `risk_kill_switch.py`：极端行情保护

### 推荐优先验证的 6 组风控实验
1. 固定止损 vs ATR 止损。
2. 无择时 vs 市场状态择时（指数均线/波动率阈值）。
3. 固定仓位 vs 波动率目标仓位。
4. 无行业约束 vs 行业中性约束。
5. 日频调仓 vs 周频调仓（比较换手与净值稳定性）。
6. 单一风控 vs 多层风控叠加（检测过拟合风险）。

## 7. 评估指标体系（不仅看收益）

建议分三层指标：

1. **收益风险层**：年化收益、波动率、夏普、Sortino、最大回撤、Calmar。
2. **交易行为层**：换手率、持仓集中度、胜率、盈亏比、持有期分布。
3. **稳健性层**：
   - 参数扰动敏感性；
   - 子区间表现一致性；
   - 不同市场风格下的绩效衰减；
   - 成本上调压力测试（佣金/滑点翻倍）。

硬性门槛建议：
- 样本外夏普与样本内夏普比值不得过低；
- 最大回撤在可承受区间内（结合你的资金回撤容忍度）；
- 成本压力测试后策略仍可用。

## 8. 版本治理与可复现建议

### 版本对象
- 代码版本（Git commit hash）
- 数据版本（快照日期 + 数据源版本）
- 参数版本（YAML/JSON 配置）
- 实验版本（run_id）

### 落地做法
- 每次研究必须有 `experiment.yaml`：
  - 股票池
  - 回测区间
  - 调仓频率
  - 费用参数
  - 风控参数
- 输出标准化报告：`report.html` + `metrics.json` + `trades.parquet`。
- 建立“晋级机制”：只有通过稳定性门槛的版本才能进入候选策略池。

---

## 9. 针对 50万~500万规模的实务建议

1. **容量管理**：避免过多小市值冷门票，优先中大盘与流动性更好的标的池。
2. **降低冲击成本**：周频/双周频调仓通常更友好。
3. **控制策略复杂度**：少参数、强解释性，便于长期坚持。
4. **组合化优先**：2~4 个低相关子策略组合，优于押注单一信号。
5. **先风控后收益**：优先把最大回撤压住，再追求收益提升。

---

## 10. 建议的首个 90 天执行路线图

### 第 1 个月：打底
- 完成数据管线 + 统一回测框架 + 基线策略 2 个。
- 跑通样本内/样本外评估与报告自动生成。

### 第 2 个月：风控强化
- 接入 3 类仓位管理 + 2 类止损机制 + 1 类市场状态过滤。
- 建立回撤分级降仓机制。

### 第 3 个月：版本筛选
- 做滚动窗口验证与参数稳健性测试。
- 输出“候选策略清单（含适用市场环境标签）”。

交付物目标：
- 一个可复现研究仓库；
- 一套标准化实验与报告流程；
- 至少 2 个可解释、可跟踪的候选策略版本。

---

## 11. 你可以马上开工的具体清单

1. 先定数据字段与回测接口，不要先写复杂策略。
2. 先实现一个“最朴素可运行策略 + 统一风控模板”。
3. 建立实验命名规范：`策略名_风控名_市场状态_版本号`。
4. 每周只新增一个变量（策略或风控其一），避免多变量混改。
5. 每周复盘“失效案例”，记录触发条件与改进假设。
